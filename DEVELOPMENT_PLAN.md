# DEVELOPMENT_PLAN

## 一、当前项目现状分析

### 1. `customer_hand` 当前实现了什么

`customer_hand` 目前是一个“最小可运行”的学习型客服后端雏形，已经具备以下能力：

- 基于 `FastAPI` 提供 HTTP 接口
- 支持健康检查 `GET /health`
- 支持发送消息 `POST /api/messages`
- 支持查看会话状态 `GET /api/tracker/{sender_id}/full`
- 支持重置会话 `POST /api/tracker/{sender_id}/reset`
- 具备基础的对话状态保存能力：`InMemoryTrackerStore`
- 具备基础 Flow 配置加载：`FlowLoader`
- 具备基础的 Agent 主流程：
  - 记录用户消息
  - 根据规则识别售后/物流意图
  - 尝试调用 LLM 命令生成
  - 解析命令并执行
  - 路由到 Action
  - 返回统一格式的消息
- 具备基础测试文件，说明它已经在做“可验证开发”而不是纯演示

从代码上看，`customer_hand` 更像一个“从规则系统逐步演进到 LLM Agent / Flow / Action / RAG 的教学项目”。

### 2. `llm_customer_service` 当前实现了什么

`llm_customer_service` 目前不是一个完整独立产品，更像是：

- 一套更完整、更重的教学工程骨架
- 包含更丰富的模块拆分、文档、示例数据、训练、检索、NLG、策略、图式编排、CLI、Web API 等内容
- 已经包含大量可复用的成熟结构：
  - `atguigu_ai/agent`
  - `atguigu_ai/dialogue_understanding`
  - `atguigu_ai/policies`
  - `atguigu_ai/retrieval`
  - `atguigu_ai/nlg`
  - `atguigu_ai/core`
  - `atguigu_ai/channels`
  - `atguigu_ai/api`
  - `atguigu_ai/cli`
  - `ecs_demo`
  - `笔记/`、`作业/`、`02_资料/`
- 现阶段更偏“课程实践 + 系统演示 + 训练/检索/编排实验平台”

它比 `customer_hand` 更完整，但也更重、更复杂，且包含不少面向教学/实验的代码与资源，不一定适合作为最终简历项目的直接上线版本。

### 3. 当前项目的技术栈

从两个项目综合判断，当前技术栈主要包括：

- Python 3.10+
- FastAPI
- Uvicorn
- Pydantic / Pydantic Settings
- PyYAML / ruamel.yaml
- python-dotenv
- OpenAI Python SDK
- Click（CLI）
- LangChain / LangGraph（在 `llm_customer_service` 中）
- Sentence Transformers / NumPy（嵌入与检索）
- SQLAlchemy / PyMySQL（结构预留）
- Neo4j / Neo4j GraphRAG（结构预留）
- WebSocket / Socket.IO（实时交互）
- pytest / httpx（测试）
- Jinja2（模板）
- 可能还会接入：Docker / Docker Compose / 向量数据库 / 观测日志体系

### 4. 当前项目存在的问题

#### `customer_hand` 的问题

- 功能还偏少，离“简历级智能客服系统”还有较大差距
- 代码中仍有“对象化结构”和“字典结构”混用的情况，后续扩展会越来越难维护
- LLM 调用、Prompt、命令解析、Flow、Action 的边界还可以继续收敛
- RAG、知识库、文档切分、向量检索等核心大模型能力尚未完成
- Docker 部署、环境隔离、配置管理、日志追踪还不够系统化
- 测试覆盖还不够完整，尤其是面向异常和边界场景

#### `llm_customer_service` 的问题

- 工程规模大，但当前更像“课程源码库”而不是“可交付产品”
- 目录层级较复杂，模块很多，学习成本高
- 训练、检索、图式编排、策略、多通道等内容过于分散
- 教学资料、笔记、作业、demo、模型文件混在一起，不利于形成清晰产品边界
- 很多代码更偏框架实验，不一定适合作为最终简历项目的主线
- 当前最需要的是“删繁就简、提炼主线、形成单一产品目标”

### 5. 当前代码中可以复用的部分

#### 来自 `customer_hand` 可复用的部分

- `app/agent/agent.py`：当前主对话编排逻辑，是后续重构的核心参考
- `app/actions/`：动作注册、动作基类、内置动作可直接迁移或改造
- `app/core/tracker.py`、`tracker_store.py`：会话状态管理思路可复用
- `app/core/flow_loader.py`：Flow 加载思路可复用
- `app/dialogue/`：Prompt、LLM、命令解析、命令处理、Flow 执行逻辑可复用
- `app/api/`：统一响应、错误处理、inspect 页面可复用
- `test/`：测试用例结构可复用

#### 来自 `llm_customer_service` 可复用的部分

- `atguigu_ai/shared/`：配置、常量、异常、YAML 工具
- `atguigu_ai/core/`：Tracker、Domain、Slots、Store 设计
- `atguigu_ai/dialogue_understanding/`：Flow、Command、Prompt、Processor、Stack 体系
- `atguigu_ai/policies/`：策略层设计
- `atguigu_ai/retrieval/`：检索层与 embedding 结构
- `atguigu_ai/nlg/`：响应改写、模板化生成
- `atguigu_ai/api/server.py`：FastAPI + WebSocket 的更完整服务封装
- `atguigu_ai/cli/`：命令行运行、初始化、调试、导出能力
- `ecs_demo/`：业务数据、domain、flow、action、样例资源
- `文档/笔记/作业`：适合作为学习材料和答题资料，但不建议直接纳入最终产品核心目录

### 6. 当前代码中需要重构或废弃的部分

#### 建议重构

- 统一消息请求/响应模型，避免不同入口返回结构不一致
- 将“规则意图识别 / LLM 命令 / Flow 路由 / Action 执行”拆分为明确层级
- 统一 tracker 数据模型，避免 dict 和对象混用导致接口和测试复杂化
- 把配置、Prompt、模型调用、RAG 索引、日志、异常处理集中到核心基础设施层
- 把 API 层与业务编排层彻底解耦
- 把 demo 数据、文档、教学资源和产品代码隔离

#### 建议废弃或降级为参考材料

- 大量纯教学文档、笔记、作业文件，不进入最终产品主目录
- 过于实验性的训练脚本，如果最终目标不是训练模型，而是构建应用层系统，则可降级
- 与最终产品无关的测试数据和历史残留目录
- 任何硬编码模型路径、硬编码业务数据、硬编码环境变量读取方式

## 二、最终目标项目定位

最终项目建议定位为：

> **基于大模型的智能客服系统，支持本地运行、Docker 部署、知识库问答、上下文记忆、Prompt 模板、流式输出、日志追踪、异常处理和可扩展模型接入。**

更适合作为简历描述的版本可以进一步表述为：

> **设计并实现一个面向电商售前/售后场景的智能客服系统，集成多轮对话、意图识别、RAG 知识库检索、LLM 调用封装、流式响应与统一观测体系，支持本地与容器化部署。**

这个定位的好处是：

- 既有业务场景，又有技术深度
- 能体现 LLM 应用开发工程师的完整能力链
- 面试官可以从架构、Prompt、RAG、工程化、部署、测试、异常处理等维度追问
- 不会落入“只是调 OpenAI 接口”的浅层项目描述

## 九、风险和注意事项

### 1. API Key 泄露

风险：
- 把密钥写进代码、README、镜像层或日志

应对：
- 使用 `.env`
- 加入 `.gitignore`
- 日志中脱敏
- 不把真实密钥提交到仓库

### 2. 依赖版本冲突

风险：
- LangChain、Pydantic、FastAPI、OpenAI SDK 版本变化较快

应对：
- 锁定版本范围
- 每阶段跑测试
- 不要一次性升级所有依赖

### 3. Docker 网络问题

风险：
- 容器访问宿主机服务失败
- Compose 服务名解析错误

应对：
- 使用 Compose 服务名访问依赖
- 明确端口映射
- 做健康检查

### 4. 模型接口超时

风险：
- 大模型响应慢导致接口卡顿

应对：
- 超时设置
- 重试机制
- 流式返回
- fallback 回复

### 5. RAG 检索不准

风险：
- 切分不合理、向量模型不匹配、召回噪声过大

应对：
- 优化 chunk 策略
- 做来源展示
- 增加 rerank 或阈值过滤

### 6. Prompt 不稳定

风险：
- 模型输出格式不一致
- 命令解析失败

应对：
- 模板化 Prompt
- 结构化输出
- 严格校验与兜底

### 7. 流式输出异常

风险：
- 断线、前后端协议不一致、消息乱序

应对：
- 统一流式协议
- 增加心跳与重连
- 记录消息序号

### 8. Windows 路径问题

风险：
- 相对路径在不同启动方式下失效
- 模板、数据、模型路径找不到

应对：
- 统一使用 `Path(__file__).resolve()`
- 避免依赖当前工作目录
- 在 Windows 上专门做启动验证

### 9. 教学代码过重

风险：
- 项目太大，开发者自己都讲不清

应对：
- 保留主线，删掉边缘实验内容
- 每个阶段只做一个能力点
- 为每个能力点准备“为什么这么设计”的答案
