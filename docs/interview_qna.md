# 面试问答备忘（customer_hand）

以下为围绕本项目的**高频追问**与建议答法。回答时建议遵循：结论一句 → 设计理由 → 局限或演进一句。

---

## 架构与职责

**Q：为什么要把 LLM 做成「命令生成」而不是直接让模型回复用户？**

A：客服场景需要可控的流程与数据校验。让模型输出结构化命令，由程序执行 Flow 和 Action，可以把幻觉挡在业务边界外；闲聊类才用 `chitchat` 直接出文。

**Q：API 层和业务层怎么分的？**

A：`main.py` 只做 HTTP、校验、trace 与响应组装；`Agent` 负责编排；具体 NLU/命令/Flow 在 `dialogue`，副作用在 `actions` 和 `tracker_store`。这样单测可以 mock store 和 LLM。

---

## 会话与多轮

**Q：上下文存在哪里？**

A：内存中的 tracker，按 `sender_id` 索引；包含最新消息、槽位、事件列表等。生产环境会换成持久化存储并考虑过期策略。

**Q：多轮对话和单轮问答的区别？**

A：多轮需要持续更新状态机（当前 Flow、待收集槽位），每轮输入依赖上一轮；单轮问答无状态依赖。本项目用 tracker + 历史摘要支持前者。

---

## RAG

**Q：你们的 RAG 是向量检索吗？**

A：当前 MVP 是**关键词索引** + 可选 LLM 归纳，优先保证可本地运行、易调试。演进方向是把 retriever 换成 embedding + 向量库，对上层接口保持不变。

**Q：怎么缓解检索不准？**

A：优化切分、调 top_k、加阈值或重排；产品上展示来源片段让用户可核对。关键词阶段更要规范用户问法或扩展同义词表。

**Q：检索到了但生成失败怎么办？**

A：返回兜底话术并带上文档来源信息，metadata 里保留 `matches`，保证可观测、可人工介入。

---

## Prompt 与稳定性

**Q：Prompt 怎么迭代？**

A：命令类与 RAG 类分开维护；变更后跑 `pytest` 与手工场景（退货、物流、知识问）。长期可把模板文件化并做版本号。

**Q：LLM 超时或挂了？**

A：`LLM_ENABLED=false` 时走规则；开启时异常会捕获并记事件，不阻塞整个进程返回。具体超时在 `LLMClient` 配置中（可结合代码说明）。

## 工程化

**Q：如何排查一次请求？**

A：用响应头或错误体里的 `trace_id` 串联日志；关注 `emit_llm_event` / `emit_rag_event` 打印的关键阶段。

**Q：和「只调 OpenAI 的 demo」区别在哪？**

A：有明确分层、会话状态、Flow/Action、RAG 与降级路径、统一错误模型与追踪，而不是单函数 `chat.completion`。

## 开放题

**Q：如果给你两周继续做什么？**

A：示例方向任选 1～2 个说深即可：向量检索与重排、持久化 tracker、流式 SSE、Docker Compose、鉴权与限流、评估集与命中指标。

**Q：最大的技术债是什么？**

A：诚实回答当前项：例如内存 tracker、关键词检索的语义天花板、或 dict/对象混用的 tracker 表示——并说明重构计划。
