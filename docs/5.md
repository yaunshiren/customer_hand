阶段 5 建议拆成 7 步，颗粒度比较适合新手：每一步都有明确产出，也不会一下子太复杂。

第 1 步：先区分问题类型

目标是让系统知道用户在问什么。

分成几类就够了：

政策类问题：走 RAG
订单查询：调用 query_order
物流查询：调用 query_logistics
投诉问题：调用 create_ticket
发票问题：可能先 RAG，也可能调用 create_invoice
这一阶段先不用急着接真实工具，可以先写分类逻辑。

第 2 步：实现第一批 mock 工具

先不要接真实数据库或接口，用假数据模拟业务动作。

例如：

query_order(order_id)
query_logistics(order_id)
create_ticket(category, description, user_id)
create_invoice(order_id, title)
产出是：每个工具都能单独调用，并返回结构化结果。

第 3 步：定义工具 schema

给每个工具写清楚：

工具名
工具描述
参数类型
必填字段
返回结果格式
这一步的重点是让模型知道：

什么时候该调用哪个工具
调用工具时需要哪些参数
第 4 步：写 Tool 调用路由逻辑

把用户问题和工具调用连起来。

规则可以先简单一点：

问政策 -> RAG
问订单/物流且有订单号 -> 调工具
问订单/物流但没有订单号 -> 追问
投诉 -> 创建工单
开发票且有订单号和抬头 -> 调工具
开发票流程问题 -> RAG
这一阶段完成后，就能跑通核心验收案例。

第 5 步：处理工具失败和降级

新手项目很容易忽略这一步，但面试里很加分。

要处理：

工具超时
工具返回空
参数缺失
工具异常
失败时不要让系统崩溃，而是返回友好提示，比如：

“我暂时没有查到物流信息，已经为你记录问题，稍后可转人工处理。”
同时记录 tool_trace。

第 6 步：加 Agent 安全限制

这是为了防止模型一直调用工具。

需要加这些限制：

最大工具调用次数
工具超时时间
最大重试次数
重复工具调用检测
高风险操作二次确认
比如开发票、退款、修改地址这类动作，后面都应该二次确认。

第 7 步：做验收测试和面试总结

用几条典型问题测试：

“查一下订单 10001 到哪了” -> 调 query_logistics

“我要投诉客服态度差” -> 调 create_ticket

“怎么开发票” -> 走 RAG

“订单 10001 开公司发票” -> 调 create_invoice

工具失败 -> 返回友好提示，不崩溃
最后整理成面试说法：

我把客服问题分成知识问答和业务动作两类。
政策类走 RAG，订单、物流、发票、投诉类走 Tool Calling。
工具调用有 schema 校验、超时、重试、trace 记录和最大调用次数限制，避免 Agent 死循环。
比较推荐的学习顺序是：

问题分类 -> mock 工具 -> schema -> 调用路由 -> 失败降级 -> 安全限制 -> 验收测试
这样学起来不会散，也能一步步看到系统从“会回答”变成“会办事”。
