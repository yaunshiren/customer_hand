第 1 步：先建 Intent Taxonomy

目标：先把“系统能识别哪些意图”定义清楚。

新增：

app/intent/schema.py
app/intent/taxonomy.py
data/intents/customer_intents.yml
customer_intents.yml 里先覆盖评测集里的所有类别：

S1-S17
F1-F3
C1-C2
UNKNOWN
每个 intent 至少包含：

id: S16_物流配送
name: 物流配送
type: KB_TOOL
description: 用户咨询发货、配送、物流进度、改地址、签收等问题
examples:
  - 我能改收货地址吗？已经发货了
  - 下单后多久能送到
  - 快递到哪里了
这里不要先纠结模型，先把意图体系建稳。
这是后面 intent_top1 能不能算的基础。

第 2 步：实现 IntentResult 数据结构

目标：统一分类器输出格式。

新增 app/intent/schema.py：

class IntentCandidate(BaseModel):
    intent_id: str
    confidence: float

class IntentResult(BaseModel):
    intent_id: str
    intent_name: str
    intent_type: str
    confidence: float
    candidates: list[IntentCandidate]
    reason: str | None = None
    source: str
建议 source 支持：

llm_classifier
rule_fallback
unknown
这样以后评测时可以知道 intent 是大模型判断的，还是规则兜底的。

第 3 步：做规则候选，不做最终决策

目标：规则只作为兜底或候选，避免又变回 _predict_eval_intent。

新增：

app/intent/classifier.py
里面先实现一个高精度规则候选器：

投诉、态度差、消协 -> F3_投诉吐槽
希望、建议、能不能加、支持儿童模式 -> F2_功能建议
保修、维修、保修期 -> S14_售后政策
发票、抬头 -> S17_发票会员
物流、快递、发货、改地址 -> S16_物流配送
充不进电、不开机、发烫、故障 -> F1_故障报告
注意：这一步不要让规则直接控制最终路由。
规则的作用是：

LLM 可用：作为候选和 fallback
LLM 不可用：作为 rule_fallback
这样它不是评测作弊，而是工程降级能力。

第 4 步：接入 LLM Intent Classifier

目标：让大模型基于 intent tree 输出结构化分类结果。

新增：

app/intent/prompt.py
Prompt 不要让模型自由发挥，只让它输出 JSON：

{
  "intent_id": "S16_物流配送",
  "confidence": 0.86,
  "candidates": [
    {"intent_id": "S16_物流配送", "confidence": 0.86},
    {"intent_id": "S15_退换货", "confidence": 0.42}
  ],
  "reason": "用户询问已发货后是否能改地址"
}
分类器流程：

加载 taxonomy
  -> 构造 prompt
  -> 调 LLM
  -> JSON 解析
  -> 校验 intent_id 是否存在
  -> 校验 confidence 是否在 0-1
  -> 成功：source=llm_classifier
  -> 失败：使用 rule_fallback
建议第一版：

temperature = 0
top_p = 1
保证输出稳定。

第 5 步：实现 Route Policy，接管旧 route

目标：把“识别意图”和“决定走哪条链路”拆开。

新增：

app/intent/policy.py
输入：

intent_result
message
tracker
输出：

{
  "execution_route": "rag",
  "system_route": "kb_tool",
  "reason": "物流政策问题，先走 RAG，再引导订单查询"
}
这里建议区分两个概念：

execution_route：代码实际走哪个节点，如 rag / ticket / flow / chitchat
system_route：业务语义路由，如 kb / kb_tool / ticket / chitchat
第一版策略可以这样：

S1-S14/S17 -> execution_route = rag

S15_退换货:
  如果只是问政策 -> rag
  如果明确申请退货且缺订单号 -> rag，先答申请规则，再引导订单号
  后续再升级成 kb_tool 或 flow

S16_物流配送:
  如果问发货/改地址/配送时效 -> rag
  如果提供订单号查物流 -> tool 或 flow
  第一版没有工具时先 rag

F1_故障报告:
  execution_route = rag
  先给基础排查步骤，再引导创建工单
  不要直接要求订单号

F2_功能建议:
  execution_route = ticket 或 chitchat
  不走 RAG

F3_投诉吐槽:
  简单投诉 -> ticket
  投诉里包含物流/故障事实 -> rag 或 ticket_after_rag
  第一版可先 ticket，但不能直接 flow 要订单号

C1_寒暄问候:
  chitchat

C2_越界提问:
  chitchat/refuse
  不走 RAG
这一步会直接解决：

F2/F3 误检索
C2 越界误检索
F1/S16 直接要求订单号
第 6 步：接入现有 Graph，但不要大拆

目标：低风险接入，不要一次性重写 Agent。

当前链路是：

load_context -> understand -> route -> branch -> generate_response -> save_context
建议改成：

load_context
  -> understand：新增 intent_result
  -> route：优先使用 route_policy，而不是 LLMCommandGenerator 的命令
  -> branch
  -> generate_response：补 metadata
  -> save_context
也就是：

understand 里调用 IntentClassifier.classify(...)。
把结果放到 state：
state["intent_result"] = intent_result
route 里优先使用 RoutePolicy.decide(...)。
旧的 LLMCommandGenerator 先保留，用作兼容：
active_flow 中继续处理 slot。
chitchat 回复暂时还能用。
后续再逐步拆薄。
metadata 里加：

{
  "intentLeafIds": ["S16_物流配送"],
  "intentSource": "llm_classifier",
  "intentConfidence": 0.86,
  "system_route": "kb_tool",
  "route": "rag"
}
这样评测侧就能计算 intent_top1。

最推荐的执行顺序

可以按这个顺序做：

1. taxonomy + schema
2. rule fallback
3. LLM classifier
4. route policy
5. graph 接入
6. metadata 评测适配
7. 测试和 system eval 回归
重点提醒一句：
这一阶段不要同时做 query rewrite、混合检索、tool calling、持久化。那些都很诱人，但会把问题搅在一起。阶段 1 的目标就一个：让系统真的知道用户在问什么，并且按稳定规则走对路。
