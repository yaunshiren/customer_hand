# Tool Calling 验收与面试总结

## 验收用例

| 用户问题 | 期望路由 | 期望结果 |
| --- | --- | --- |
| 查一下订单 10001 到哪了 | Tool Calling | 调用 `query_logistics` |
| 我要投诉客服态度差 | Tool Calling | 调用 `create_ticket` 创建工单 |
| 怎么开发票 | RAG | 不调用工具，走知识库回答 |
| 订单 10001 开公司发票 | 二次确认后 Tool Calling | 先要求确认，用户确认后调用 `create_invoice` |
| 工具失败 | 降级回复 | 返回友好提示，不让对话崩溃 |

说明：第 6 步已经加入高风险操作二次确认，所以开票不是直接执行，而是先保存待确认工具调用。用户回复“确认”后，系统再真正调用 `create_invoice`。


推荐运行：

## 面试讲法

我把客服问题拆成两类：知识问答和业务动作。

政策、流程、规则类问题走 RAG，例如“怎么开发票”“退货规则是什么”。这类问题只需要从知识库检索并生成回答。

订单、物流、投诉、开票这类问题走 Tool Calling，因为它们不是单纯回答知识，而是要执行业务动作。比如“查一下订单 10001 到哪了”会调用 `query_logistics`，“我要投诉客服态度差”会调用 `create_ticket`。

为了保证工具调用稳定，我做了几层工程化处理：

- 每个工具都有 schema，校验工具名、参数类型和必填字段。
- 工具返回统一封装为 `ToolCallResult`，成功和失败都有结构化结果。
- 工具有超时、最大重试、空结果处理和异常捕获。
- 每次工具调用都会记录 `tool_trace`，方便排查问题。
- Agent 层限制最大工具调用次数，并检测重复工具调用，避免模型死循环。
- 对开票这类高风险动作做二次确认，用户确认后才真正执行。

一句话总结：

> RAG 解决“知道什么”，Tool Calling 解决“能做什么”。我把客服问题按这两类路由，并在工具调用层加了 schema 校验、超时重试、失败降级、trace 记录、最大调用次数和高风险二次确认，保证系统既能办事，也不会失控。

## 学习顺序

推荐按这个顺序学：

```text
问题分类 -> mock 工具 -> schema -> 调用路由 -> 失败降级 -> 安全限制 -> 验收测试
```

这个顺序的好处是，每一步都有明确产出：

- 问题分类：知道用户要问知识，还是要办业务。
- mock 工具：先不用接真实接口，也能跑通业务动作。
- schema：让工具参数变得可校验、可解释。
- 调用路由：把用户问题和工具调用连起来。
- 失败降级：工具出错时不崩溃。
- 安全限制：防止重复调用、无限调用和高风险误操作。
- 验收测试：用典型问题证明系统从“会回答”升级为“会办事”。
