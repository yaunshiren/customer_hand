建议把阶段 6 拆成 4 个子阶段，粒度比较合适：既能覆盖 Memory、Rewrite、检索接入、验收测试，又不会拆得太碎。

阶段 6.1：Memory 数据结构与上下文管理
目标：先把“记住什么”定义清楚。

内容：

保留最近 N 轮对话作为短期记忆。
设计实体记忆结构，例如产品、订单号、问题类型。
定义长历史摘要字段，先预留接口。
trace 中记录当前 memory 状态。
产出：

{
  "recent_turns": [],
  "memory_entities": {
    "product": "",
    "order_id": "",
    "intent": ""
  },
  "summary": ""
}
阶段 6.2：实体抽取与记忆更新
目标：从用户问题和助手回答中更新上下文。

内容：

从多轮对话中识别最近提到的产品、订单号、服务类型。
新实体覆盖旧实体，未提到则继承最近实体。
支持至少这些实体：
product
order_id
issue_type / intent
示例：

用户：小米 14 Pro 保修多久？
更新：

{
  "product": "小米 14 Pro",
  "intent": "保修"
}
阶段 6.3：Query Rewrite 改写模块
目标：把当前问题改写成可独立检索的问题。

内容：

输入当前 query、最近对话、实体记忆。
输出 original_query、rewritten_query、memory_entities。
处理指代词，例如“它”“这个”“刚才那个”“那款”。
对不需要改写的问题保持原样。
示例：

{
  "original_query": "那它可以 7 天无理由吗？",
  "rewritten_query": "小米 14 Pro 可以 7 天无理由退货吗？",
  "memory_entities": {
    "product": "小米 14 Pro"
  }
}
阶段 6.4：检索接入与验收测试
目标：确保 rewritten query 真正参与 RAG 检索，并且 trace 可见。

内容：

检索时使用 rewritten_query，而不是 original_query。
trace 中展示：
original_query
rewritten_query
memory_entities
是否触发 rewrite
准备至少 3 个多轮样例测试。
验收样例可以包括：

产品指代：
用户：小米 14 Pro 保修多久？
用户：那它可以 7 天无理由吗？
订单指代：
用户：订单 123456 到哪了？
用户：它还能改地址吗？
问题类型延续：
用户：华为 Mate 60 支持退货吗？
用户：那换货呢？
最终建议写成：

阶段 6 拆为 4 个子阶段：
1. Memory 数据结构与上下文管理
2. 实体抽取与记忆更新
3. Query Rewrite 改写模块
4. 检索接入与验收测试
这个拆法比较稳，2 天内也好排期：前两项偏基础，第三项是核心能力，第四项保证真正闭环。
