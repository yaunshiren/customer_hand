阶段 3 可以分成 6 步 做。这个粒度比较适合新手：每一步都有明确产物，也不会一上来就把 RAG 全部推翻重写。

第 1 步：先做检索诊断，不急着改代码

目标是知道现在到底错在哪里。

要做：

对最新 eval 报告拆分 badcase。
找出 hit@1 = 0、recall@3 低的样本。
重点看 S6_配件兼容、S14_售后政策、S9_配网连接、S13_保养维护、F1_故障报告。
对每条 badcase 记录：
用户问题
期望 doc_id
实际 retrieved_doc_ids
错因：实体没命中、意图错、文档排序错、知识库缺失、chunk 太碎/太粗
产物：

docs/rag_retrieval_badcases.md
这一阶段不要凭感觉优化，先让错误变得可见。

第 2 步：补齐文档 metadata 和检索字段

目标是让后面的 intent filter、entity boost、rerank 有数据可用。

每个 chunk/doc 最好有：

{
  "doc_id": "POLICY_WAR_001",
  "title": "小米手机保修政策",
  "intent_ids": ["S14_售后政策"],
  "product_names": ["小米 14 Pro", "Redmi K70"],
  "category": "policy",
  "keywords": ["保修", "维修", "进水", "碎屏"]
}
要做：

检查当前知识库文档是否已有 doc_id/title/category。
给重要文档补 intent_ids、product_names、keywords。
优先补：
售后政策
退换货政策
物流政策
配件兼容
操作指引
故障排查
产物：

data/knowledge/... 文档 metadata 更完整
test/test_document_metadata.py 补测试
这一步很重要，因为混合检索不是只写算法，也要让文档有可检索的结构信息。

第 3 步：实现 Hybrid Retriever 骨架

目标是保留原来的接口，但内部支持多通道召回。

建议新增：

app/rag/hybrid_retriever.py
app/rag/scoring.py
保留上层调用不变：

knowledge_answerer.answer(query, top_k=3)
内部先做：

keyword search
vector search 或现有 retriever
intent-directed search
merge
dedup
第一版不用做得很复杂，可以这样：

keyword search：标题、正文、keywords 命中加分。
vector search：复用现有向量或现有 retriever。
intent-directed search：如果 metadata 里 intent_ids 包含当前 intent，就额外召回或加分。
merge：按 doc_id + chunk_id 去重。
产物：

HybridRetriever.retrieve(query, intent_id=None, top_k=3)
这一阶段目标是“能跑通多路召回”，不是一次性追求最高分。

第 4 步：做轻量 Rerank

目标是把召回到的候选重新排序，让正确文档尽量排到 top1。

建议新增：

app/rag/reranker.py
第一版规则可以很朴素：

intent_id 匹配       +0.25
产品型号完全匹配    +0.30
标题命中关键词      +0.20
正文命中关键词      +0.10
政策词命中          +0.15
chunk 太短          -0.10
chunk 太长          -0.05
重点优化：

小米 14 Pro 用什么充电器 应该优先命中配件/充电器文档。
保修期多久 应该优先命中售后政策文档。
已经发货能改地址吗 应该优先命中物流政策文档。
滤芯多久换 应该优先命中保养维护文档。
产物：

RuleBasedReranker.rerank(query, candidates, intent_id=None)
test/test_rag_reranker.py
这一步是面试里最好讲的：不是简单 topK，而是结合业务信号重排。

第 5 步：实现 Context Builder 和 Citation

目标是让回答生成拿到更清晰的上下文，也让评测 metadata 更完整。

建议新增：

app/rag/context_builder.py
app/rag/citation.py
上下文格式建议变成：

[来源 1]
doc_id: POLICY_WAR_001
chunk_id: POLICY-WAR-001-003
title: 小米手机保修政策
content: ...

[来源 2]
...
metadata 中稳定输出：

{
  "rag_doc_ids": ["POLICY_WAR_001"],
  "rag_chunk_ids": ["POLICY-WAR-001-003"],
  "rag_context_doc_ids": ["POLICY_WAR_001"],
  "retrieved_contexts": ["..."]
}
产物：

ContextBuilder.build(matches)
CitationBuilder.from_matches(matches)
这一步会让回答更可控，也方便之后做 RAGAS 或 LLM-as-judge。

第 6 步：跑评测，按指标迭代

目标是证明混合检索真的有效。

先跑单元测试：

pytest
再跑 RAG/system eval：

python -m eval rag run --target customer_hand --customer-mode system
python -m eval rag score eval/runs/xxx.jsonl --skip-ragas
重点看：

hit@1
hit@3
recall@3
mrr@10
S6 top1
S14 top1
S16 top1
目标可以先定得现实一点：

指标	当前	第一轮目标
hit@1	78.0%	85%+
hit@3	90.2%	93%+
mrr@10	83.7%	88%+
S14 top1	偶有失误	稳定
S6 top1	偶有失误	稳定
等第一轮稳定后，再冲：

hit@1 90%+
hit@3 95%+
mrr@10 90%+
推荐执行顺序就是：

1. 检索 badcase 诊断
2. 补 metadata
3. Hybrid Retriever 骨架
4. 规则 Rerank
5. Context Builder + Citation
6. pytest + system eval 复测
这套顺序面试也很好讲：先诊断，再补结构化信息，再做多路召回，再重排，最后用指标证明优化有效。
